# cvkit — Academic CV Management CLI

## What This Is

CLI and library for aggregating academic/professional data from multiple sources (ORCID, Semantic Scholar, GitHub, LinkedIn PDF) into a canonical `cv.json` file. This is the **data engine** in a three-project architecture:

- **cvkit** (this repo, data engine) → fetches, merges, validates, builds cv.json
- **icv** ([github.com/AlexanderPico/icv](https://github.com/AlexanderPico/icv)) → static Astro site that renders cv.json
- **cvkit-app** (not yet built) → SvelteKit web app GUI for non-technical users

cvkit is designed to be consumed three ways:
1. **CLI** — `cvkit fetch`, `cvkit build`, etc. (power users)
2. **Library** — imported by cvkit-app to run fetch pipelines server-side
3. **npm packages** — `@cvkit/schema`, `@cvkit/store`, `@cvkit/ingesters`, `@cvkit/cli`

## CVData Schema

Extends [JSON Resume](https://jsonresume.org/schema/) with academic fields. Key types in `packages/schema/src/index.ts`:

- `CVData` — root type. All fields optional.
- `Basics` — name, label, email, url, summary, location, profiles, orcidId
- `Publication` — title, type, authors[], doi, pmid, s2PaperId, citationCount, year, journal
- `Grant` — title, funder, awardNumber, role (PI/co-PI), dates, amount
- `Software` — name, description, githubRepo, language, stars, forks, topics
- `CitationMetrics` — hIndex, totalCitations, i10Index, publicationCount, s2AuthorId
- `WorkEntry`, `EducationEntry` — standard JSON Resume
- `FetchMeta` — timestamps per source
- `IngestResult` — Partial<CVData> returned by ingesters
- `IngestResponse` — { source, ok, data?, error?, fetchedAt } envelope

## Two-Layer Data Model

- `cv.json` — auto-generated by `cvkit fetch`. Overwritten on each fetch.
- `cv.overrides.json` — manual corrections that **always win** and survive re-ingestion. Merged on top of cv.json by the store.
- Deduplication: publications are deduped by DOI, then by normalized title.

## CLI Commands

```bash
cvkit init [--overwrite]                    # Create empty cv.json
cvkit fetch --orcid <id>                    # Fetch ORCID data
cvkit fetch --github <username>             # Fetch GitHub repos
cvkit fetch --s2-orcid <id>                 # Fetch Semantic Scholar via ORCID
cvkit fetch --s2-author <id>                # Fetch Semantic Scholar by author ID
cvkit fetch --linkedin-pdf <path>           # Parse LinkedIn PDF export
cvkit fetch --all                           # Run all configured sources
cvkit validate                              # Check cv.json structure
cvkit diff --orcid <id>                     # Preview changes without writing
cvkit build --format json|markdown|text     # Render cv.json to format
cvkit build --format json -o path/cv.json   # Write to specific path
```

### Future Ingesters
The umbrella project goal is "gathering online identity information." Current coverage: ORCID, Semantic Scholar, GitHub, LinkedIn PDF. Natural additions:
- **Google Scholar** — broader citation data, especially for non-biomedical fields
- **Personal website scraper** — extract bio, publications, contact info
- **Social profiles** — Twitter/X, Bluesky, Mastodon bios and links
- **Public records APIs** — depending on use case

Each new ingester follows the same pattern: implement an async class returning `IngestResponse`, register it in `packages/ingesters/src/index.ts`, add a CLI flag in `packages/cli/src/cli.ts`.

### cvkit-app (Next Major Project)
Architecture already decided:
- **Stack**: SvelteKit + Cloudflare Pages + Workers Unbound
- **Storage**: Cloudflare KV (cache) + GitHub fork (canonical — cv.json committed to user's icv fork)
- **Auth**: ORCID/GitHub OAuth via @auth/sveltekit
- **Why Workers Unbound**: LinkedIn PDF parse can be slow; standard Workers Bundled has 30s CPU ceiling
- **Why not Supabase**: Git history on the icv fork is version control; KV is just a read cache
- **Build order**: 6 phases, each independently shippable — fetch pipeline → editor → deploy → auto-refresh → polish
- **Key integration point**: cvkit-app calls cvkit as a library, stores result to KV + commits cv.json to user's icv fork, triggers GH Actions rebuild
